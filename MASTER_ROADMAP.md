### 決策 #004: 模型從 Qwen3.5-2B 升級至 Qwen3.5-4B
**日期**: 2026-03-07
**決策者**: 專案負責人
**決策內容**: 客戶端推理模型從 Qwen3.5-2B 改為 Qwen3.5-4B (官方 Ollama, 3.4GB)
**原因**:
- 2B 模型經測試確認不堪用 (bias_score 完全錯誤, spread 極大)
- 4B 模型搭配上下文注入後準確度顯著提升
- 4B 推理速度可接受 (6-10s/篇, think=false)
- 2B vs 4B 大小差異僅 ~1.4GB (2GB→3.4GB)
**測試數據**:
- 2B 無上下文: bias=95 (預期 18), 完全失敗
- 4B+RAG: bias_err 從 45 降至 7 (改善 84%)
- 最佳配置: think=false, temperature=0.5, top_p=0.95
**影響範圍**: T03 (模型配置), T04 (下載大小提示), T07 (效能基準)
**狀態**: ✅ 已執行

### 決策 #005: 採用中央空廚架構
**日期**: 2026-03-07
**決策者**: 專案負責人
**決策內容**: 爬蟲後端獨立為閉源 GitHub 項目,採用「中央空廚」架構統一爬取與處理新聞
**原因**:
- 統一爬取可確保數據品質和合規性
- 集中處理嵌入和索引,客戶端無需承擔此計算
- 後端 CPU 即可運行 (embedding + indexing 不需 GPU)
- 閉源保護爬蟲策略和知識庫
**架構**:
- 爬蟲後端: 爬取→bge-small-zh 議題篩選→清洗→推送 PowerReader API
- PowerReader: Workers 接收→bge-m3 嵌入→Vectorize 知識查詢→客戶端組裝 Prompt→本地推理
**影響範圍**: T01 (API設計), T02 (爬蟲架構), T03 (Prompt組裝), T04 (API串接)
**狀態**: ✅ 已執行

### 決策 #006: 採用 RAG 三層 Prompt 架構 + 知識透明化
**日期**: 2026-03-07
**決策者**: 專案負責人
**決策內容**: 推理 Prompt 分為三層 (核心靜態/RAG動態/輸入),知識注入對使用者透明
**原因**:
- 測試驗證: 小模型需要更多上下文而非更少 (反直覺)
- RAG 動態注入比全塞更高效 (信噪比更高)
- 使用者可展開查看注入的背景知識 (建立信任)
- 知識庫可獨立擴展,不影響 Prompt 結構
**RAG 層級**:
- L1 (靜態, ~300 tokens): 立場定義+分數錨點+JSON格式
- L2 (動態, ~200-800 tokens): 政治人物/媒體/議題/名詞/事件
- L3 (輸入): 新聞原文+輸出指令
**影響範圍**: T03 (Prompt設計), T01 (API), T04 (透明化UI)
**狀態**: ✅ 已執行

### 決策 #007: 嵌入模型選型 — bge-m3 + bge-small-zh
**日期**: 2026-03-07
**決策者**: 專案負責人
**決策內容**:
- 知識查詢嵌入: Cloudflare Workers AI `@cf/baai/bge-m3` (1024d, 邊緣 GPU, 免費)
- 議題篩選嵌入: `bge-small-zh-v1.5` (512d, CPU, Crawler 端)
- 移除 CKIP BERT (不再需要分詞)
- 移除 MiniLM (被 bge-m3 取代)
**原因**:
- bge-m3 支援中文 (100+ 語言), Cloudflare 原生整合
- bge-small-zh 輕量 (130MB), CPU 上 0.1s/篇, 適合 GitHub Actions
- CKIP BERT 的分詞功能在 RAG 架構中不再必要
- 每篇嵌入僅消耗 ~1.6 neurons, 日消耗 960/10,000 (9.6%)
**影響範圍**: T02 (移除 CKIP), T03 (嵌入模型), T01 (Vectorize 整合)
**狀態**: ✅ 已執行

### 決策 #008: Cloudflare 全棧儲存分工
**日期**: 2026-03-07
**決策者**: 專案負責人
**決策內容**: PowerReader 使用 Cloudflare 全棧:
- R2: 文章全文儲存 (永久累積, ~2.2GB/年)
- D1: 結構化資料 (使用者/分析/獎勵/文章索引)
- Vectorize: 知識庫向量搜索 (bge-m3 1024d)
- KV: 系統設定 + 快取 (少寫多讀)
- Workers AI: bge-m3 嵌入
- Pages: PWA 前端
**原因**:
- KV 寫入限制 1,000/天,不適合做主要儲存
- R2 出流量免費,適合客戶端下載
- D1 支援 SQL 查詢,5M 讀/天
- Vectorize 原生向量搜索,免自建索引
- 所有服務均在免費額度內
**影響範圍**: T01 (架構設計), 所有團隊
**狀態**: ✅ 已執行

### 決策 #009: 議題篩選採用 bge 相似度方案
**日期**: 2026-03-07
**決策者**: 專案負責人
**決策內容**: Crawler 使用 bge-small-zh-v1.5 cosine 相似度篩選社會/政治議題
**原因**:
- 比 NLI 分類器更穩定、更透明、更快 (0.1s vs 0.5s)
- 不需要微調模型,只需定義主題向量 + 校準閾值
- 新增/刪除主題向量不影響其他分類
- 測試後如準確度不足再評估微調需求
**備選方案**: 如 bge 相似度不足 → 評估 mDeBERTa NLI 或微調模型
**影響範圍**: T02 (篩選邏輯)
**狀態**: ✅ 已執行

### 決策 #010: Crawler 執行頻率每 2 小時
**日期**: 2026-03-07
**決策者**: 專案負責人
**決策內容**: GitHub Actions cron 每 2 小時執行, 每次處理 ~50 篇 (去重後), 日產 ~600 篇
**原因**:
- 每小時: Actions 分鐘超額 + Vectorize 查詢超額
- 每 4 小時: 安全但時效性降低
- 每 2 小時: Actions ~1,620 min/月 (81%), Vectorize ~18.4M/月 (61%), 最佳平衡
**影響範圍**: T02 (cron 設定), T07 (監控)
**狀態**: ✅ 已執行
