## 特色

- **資訊去中心分析** — AI 模型直接在你的瀏覽器中運行，可以直接透過你的GPU代理分析最終結果共享全網
- **公民算力驅動** — 不靠廣告、不靠企業、不由任何政治立場主導。每位使用者貢獻自己的 GPU 算力，共同建構分析生態
- **三層透明機制** — 每筆分析都可展開查看：AI 使用了什麼評分框架、補充了哪些背景知識、原始輸入是什麼。附推論指紋供驗證，難篡改。
- **跨媒體盲區偵測** — 不只告訴你這幾篇偏(共同類似新聞)偏向哪邊，還自動偵測哪些事件只有單一陣營在報導、哪些觀點被集體忽略
- **社群資料庫** — 808+ 筆政治人物、社會議題、歷史事件資料，任何人可建議編輯，審核後合併。(第一次，初始資料是AI生成，可能並不完全準確，需要要維護)

## 概念

臺灣媒體報導與內容詮釋內容往往都被披上了一層外衣，一事件經過不同媒體的詮釋後，讀者看到的面貌往往截然不同。演算法推薦又加深了「同溫層」效應 — 你接收到的觀點，可能只是整個光譜的一小段。媒體識讀向來都是一個有挑戰性的社會議題，再開發這項專案之前，我實際做了大量的研究調查。如果一個人能夠看到單個事情的多面角度，可以大幅降低對反面陣營的敵對心理，並且內容基於理性事實角度出發，實際上是有助於回歸了解事件全貌的公民角色，而不是基於顏色出發的跟隨者。PowerReader的概念幾乎是站立在"偏見分析"的事實上執行訊息清洗，讓不是自然人的AI完成些事情，並且可以很輕鬆看見整個運算的根據和透明性，也同時讓任何參與公民都是有價值的。

PowerReader 的目標是**讓你看見完整的光譜**：這篇報導偏向哪個陣營？偏了多少？同一件事不同媒體怎麼說？哪些事件只有單一立場的媒體在報導？

與傳統新聞分析工具不同，PowerReader 的分析不是由某個機構或編輯台產出的 — 它由使用者社群用自己的裝置算力集體完成，所有分析邏輯開源、可溯源、可驗證。
因為任何組織在同溫層的眼裡都會被貼上特定標籤，這幾乎是不可避免的，從最初使的製作就決定不仰賴任何組織支持，我本身也沒有任何背景，所以PowerReader的整個開發過程都是由我本人和claude獨立製作完成。

## 開發問題
       1.PowerReader這名稱挺挺鳥的，我自己是沒問過Claude意見，然後開源軟體也一堆都叫這名稱XD，搞不好未來有個社群是可以討論改改看的
       2.Qwen3 8B模型是不得已的，我預估是未來會有Qwen3.5 4B可以使用，降低門檻，提高品質；LLM模型需要轉譯成瀏覽器可用，但我本身沒有足夠算力和資源，我只能等WebGPU的開源社群提供
       3.臺灣大語言模型問題同上，根本問題是沒有這麼小的蒸餾的模型，轉譯也是需要大量資源和算力，最首先的是沒有這種臺灣大語言模型在這種小參數能推理的
       4.代碼安全性是可行的，但網域和伺服器都是基於cloudflare免費提供，安全固然有限，GPU的一切運作必然只會是你開啟網頁才運作，並且不會下載任何檔案。
       5.全網共享這套理念是可以被實踐的，網站指是開頭，基於網站使用情況和人數支持下，會持續開發瀏覽插件、Line可相容的新聞閱讀插件等等，或著其它領域
       6.內容盲測，"事實查核"是可能被統計學實現的，保守成功率落在60~70%左右，但最後並沒有實裝，因為現有模型能力/智性不足，成本太高了，因此目前只有做最簡單的偏見和立場分類。

### 2. 觀察偏見

<!-- TODO: 截圖 — 跨媒體比較頁面，同一事件多家媒體的立場分歧視覺化 -->

#### 跨媒體比較
選擇一個事件，並排比較 20+ 家台灣網路媒體的報導角度。系統計算「立場分歧分數」，一眼看出各家媒體在同一事件上的偏向差異。
(媒體的新聞內容是先被篩選的，針對社會新聞、政治議題被優先選入，娛樂類目前是被排除的)

涵蓋的媒體包括：中央社、自由時報、聯合報、公視、風傳媒、三立、ETtoday、TVBS、東森、科技新報、報導者、關鍵評論網等。

#### 分析透明化

<!-- TODO: 截圖 — 展開「分析依據」三層面板 -->

分析完成後，你可以展開「查看分析依據」查看三層透明資訊：
1. **系統分析框架** — AI 使用的評分規則（0=泛綠 ~ 100=泛藍）
2. **背景知識注入** — 本次分析時注入給 AI 參考的知識庫條目
3. **原始文章** — 實際送入模型的文章內容（截取前 8,400 字）

每筆分析附有推論指紋（模型版本、Prompt 雜湊、GPU 資訊、時間戳），可供第三方驗證。



### 4. 立場光譜怎麼看

```
偏綠 ◄──────────── 中立 ────────────► 偏藍
0        25        50        75       100
```

**陣營比例**反映「誰的聲音在這篇報導中主導」：

| 陣營 | 意義 |
|------|------|
| 綠 | 民進黨及泛綠陣營的論述佔比 |
| 藍 | 國民黨及泛藍陣營的論述佔比 |
| 白 | 民眾黨及泛白陣營的論述佔比 |
| 灰 | 中立、非政治性內容佔比 |

**情緒煽動度**（0–100）內容用字遣詞的激烈程度：

| 區間 | 等級 |
|------|------|
| 0–20 | 冷靜客觀 |
| 21–40 | 略帶立場 |
| 41–60 | 情緒化 |
| 61–80 | 煽情 |
| 81–100 | 極端煽動 |

非政治新聞的立場分數固定為 50（中立），陣營比例全為灰色。

## 知識庫

<!-- TODO: 截圖 — 知識庫頁面，顯示政治人物/議題條目與各黨立場比較 -->

知識庫是 PowerReader 分析品質的基礎。AI 分析報導時，會自動從知識庫中抓取相關的政治人物背景、議題脈絡、歷史事件，作為判斷依據注入。

### 瀏覽與查詢
- 依類型篩選：政治人物 / 社會議題 / 歷史事件
- 依政黨篩選：國民黨 / 民進黨 / 民眾黨 / 時代力量 / 台灣基進
- 關鍵字搜尋

### 各黨立場比較
每筆議題條目都可展開「各黨立場比較」，並排查看民進黨、國民黨、民眾黨對同一議題的官方立場。

### 社群編輯
任何人都可以對知識庫條目建議修改：
1. 點擊「建議編輯」，修改內容並填寫修改原因
2. 系統自動建立 Pull Request
3. 管理者審核通過後合併，網站自動更新

**編輯規範：**
- 內容必須客觀、可查證，不可包含主觀意見
- 不可使用侮辱性或歧視性語言
- 日期格式統一 YYYY-MM-DD
- 每欄位上限 120 字

## 公民算力飛輪

PowerReader 的運作模式是一個正向循環：

```
你貢獻算力分析一篇報導
       ↓
  獲得點數獎勵（每次 0.1–0.5 點）
       ↓
  分析結果匯入全平台
       ↓
  更多人看見不同觀點
       ↓
  更多人加入 → 飛輪越轉越快
```

### 點數與配額(實驗性，目前本身沒有任何用途)
- 每日有免費分析配額，用完後仍可繼續分析（不獲得點數）
- 配額每日 24:00（台灣時間）重置
- 分析結果經四道品質驗證（格式、數值範圍、歷史一致性、重複防護）
- 連續 3 次未通過品質驗證 → 暫停分析 1 小時

### 點數商店（開發中，目前本身沒有任何用途）

點數商店被我暫時關著，同上原因：

- **外觀徽章**：公民分析師、動力貢獻者、資料先鋒
- **功能道具**：分析加量包（+10 次配額）、冷卻跳過券、歷史記錄擴展（30 天→6 個月）

## 平台維護與永續

### 贊助模式

PowerReader 不靠廣告收入，計畫透過公民贊助維持運作。贊助功能正在開發中，資金分配設計如下：

| 贊助方案 | 平台基金 | 代理運算 | 開發者 |
|---------|---------|---------|--------|
| 請杯咖啡 | — | — | 100% |
| 公民力量 | 80% | — | 20% |
| 算力推動 | 40% | 50% | 10% |
| 代理媒體 | — | 80% | 20% |

- **平台基金**：建構點數商店與獎勵機制
- **代理運算**：為沒有高效能裝置的使用者提供伺服器端 AI 分析
- **開發者**：維護開發

### 開源與透明
- 所有程式碼以 [AGPL-3.0](LICENSE) 授權公開
- 分析邏輯（prompt 設計、評分框架）完全可追溯
- 獎金池抽獎機制去中心化 — 開發者只能設定每月配額，無法介入結果

### 社群共建
- 知識庫由社群持續更新，審核合併流程公開透明
- 分析結果由全體使用者集體GPU產出，非單一機構判定；關於代理運算，是直接採用Cloudflare的AI算力進行支持。
- 歡迎在 [Issues](https://github.com/MackinHung/PowerReader/issues) 回報問題或提出建議

## 隱私承諾

- 報導全文**只在你的裝置上處理**，不上傳至任何伺服器
- 提交至平台的僅有結構化分析結果（分數、比例、摘要），不含原文
- 離線時分析結果暫存本地，恢復連線後自動同步
- 所有核心功能無需登入即可使用
- 支援資料匯出與帳號刪除

## 技術架構

```
Crawler
  → RSS/API 爬取 → bge-small-zh 議題篩選 → 去重清洗
  → POST /api/v1/articles/batch → PowerReader API

PowerReader API
  → bge-m3 嵌入 (Workers AI) → Vectorize 知識搜索
  → R2 (全文) + D1 (結構化資料) + KV (設定/快取)

瀏覽器端（Cloudflare Pages PWA）
  → 取得文章 + 知識 → 組裝三層 Prompt
  → WebLLM Qwen3-8B 本地推論（WebGPU）→ 結果 + 透明化面板
```

| 層級 | 技術 |
|------|------|
| 前端 | SvelteKit 2 + Svelte 5 + TypeScript |
| AI 推論 | WebGPU + [WebLLM](https://github.com/mlc-ai/web-llm)（MLC） |
| 語言模型 | [Qwen3](https://github.com/QwenLM/Qwen3)-8B（4-bit, 4.3GB） |
| 後端 | Cloudflare Workers + D1 + R2 + KV + Vectorize |
| 嵌入模型 | Workers AI bge-m3（1024d）+ bge-small-zh（512d） |
| 向量搜索 | Cloudflare Vectorize |
| 部署 | Cloudflare Pages + Workers |
