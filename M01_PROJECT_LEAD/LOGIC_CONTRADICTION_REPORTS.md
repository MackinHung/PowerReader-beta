## 🔴 範例: 矛盾 #001 (已解決)

### 矛盾 #001: CKIP BERT 輸出欄位名稱不一致
**日期**: 2025-03-06 (範例日期)
**發現者**: M01
**優先級**: 🔴 高
**狀態**: ✅ 已解決

**問題描述**:
T02 (數據獲取團隊) 使用 CKIP BERT 輸出 `words` 欄位,但 T01 (系統架構團隊) 的 KV Schema 定義為 `tokens` 欄位。這會導致 T02 → T03 的資料傳遞失敗。

**涉及團隊**:
- T01: 定義了 KV Schema 使用 `tokens` 欄位
- T02: CKIP BERT 輸出使用 `words` 欄位
- T03: 期望接收 `tokens` 欄位

**影響範圍**:
- T02 寫入 KV 的資料格式與 T01 Schema 不符
- T03 讀取 KV 時會找不到 `tokens` 欄位,導致分析流程中斷
- 可能導致系統完全無法運作

**建議方案**:
1. **統一使用 `tokens` 欄位** (符合 NLP 業界慣例)
   - 優點: 符合業界標準,易於理解
   - 缺點: T02 需要修改 CKIP BERT 輸出格式
2. 統一使用 `words` 欄位
   - 優點: T02 不需要修改
   - 缺點: 不符合業界標準
3. 兩個欄位都保留 (不推薦)
   - 優點: 暫時兼容
   - 缺點: 未來維護困難,違反 SSOT 原則

**需要專案負責人決策的問題**:
請確認是否採用方案 1 (統一使用 `tokens` 欄位)?

**解決方案** (確定後填寫):
採用**方案 1**: 統一使用 `tokens` 欄位。

**執行步驟**:
- [x] T01 在 `KV_SCHEMA.md` 明確定義使用 `tokens` 欄位
- [x] T02 修改 CKIP BERT 輸出格式,使用 `tokens` 欄位
- [x] T03 確認讀取邏輯使用 `tokens` 欄位
- [x] M01 在 `shared/enums.js` 加入欄位名稱驗證
- [x] M01 更新 MASTER_ROADMAP.md 決策紀錄

**驗證方式**:
1. T02 輸出範例 JSON,確認包含 `tokens` 欄位
2. T03 成功讀取並解析 `tokens` 欄位
3. 端到端測試: 爬取 → 分詞 → 分析流程完整運行

**變更紀錄**:
| 日期 | 操作 | 說明 |
|------|------|------|
| 2025-03-06 | 建立 | M01 發現欄位名稱不一致 |
| 2025-03-06 | 專案負責人確認 | 採用方案 1 |
| 2025-03-06 | 解決 | 所有團隊已更新為 `tokens` |

## 🔍 常見矛盾類型 (基於 OceanRAG 經驗)

### 1. 資料結構不一致
**典型案例**: 欄位名稱不一致 (如 `tokens` vs `words`)
**預防措施**: 在 `shared/enums.js` 集中定義所有欄位名稱

### 2. 狀態機矛盾
**典型案例**: 文章狀態跳躍 (如直接從 `crawled` 跳到 `published`)
**預防措施**: 在 `shared/enums.js` 定義狀態機轉換規則

### 3. 模型版本不一致
**典型案例**: T02 使用 CKIP BERT v1.0, T03 使用 v2.0
**預防措施**: 在 `shared/config.js` 鎖定模型版本,啟動時驗證

### 4. Rate Limit 持久化問題
**典型案例**: Rate Limit 計數器存在記憶體中,重啟歸零
**預防措施**: 強制使用 KV 持久化

### 5. 錯誤訊息洩漏
**典型案例**: 將內部 traceback 顯示給 LINE Bot 用戶
**預防措施**: 在 `shared/enums.js` 定義用戶可見錯誤訊息
